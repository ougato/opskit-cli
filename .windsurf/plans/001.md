# OpsKit 运维面板 — 总体规划

## 项目定位

跨平台（Linux / Windows / macOS）商用级运维工具，单文件可执行，Powerline 风格交互菜单，全自动安装卸载 + 异常自愈 + 实时进度，解放用户搜索命令的痛苦。

---

## 技术栈

- Python 3.10+，Rich（主题 / 进度条 / 面板），Typer（CLI），psutil（系统监控），httpx（HTTP 客户端）
- 复用 cdn-rotator/tools 的 Powerline 主题体系（Catppuccin Mocha 渐变 Banner + 色块 breadcrumb）
- Nuitka / PyInstaller 打包为单文件可执行程序
- YAML 配置持久化（config/common.yaml）

---

## 主题系统架构

### 设计目标

所有文字颜色、图标、边框样式、渐变色、面板宽度等**视觉表现**完全由主题层控制。功能逻辑代码（menu.py / commands.py / prompt.py 等）**不硬编码任何颜色或样式字符串**，只通过语义化 Token 引用。换主题 = 改一个 YAML 文件或新增一个 YAML 文件，**零改动**功能代码。

### 核心理念

```
功能代码                主题层                    主题文件
─────────         ──────────────           ───────────────
menu.py           core/theme.py            themes/catppuccin.yaml
commands.py  ───→ 提供语义 Token  ←──────  themes/nord.yaml
prompt.py         print_success()          themes/dracula.yaml
progress.py       print_error()            ...
                  get_color('success')
                  get_icon('back')
```

**铁律：功能代码只调用语义 API，永远不写 `'bold green'`、`'#ff5f5f'` 等具体样式。**

### Ctrl+C 导航行为规范

| 位置 | Ctrl+C 行为 |
|---|---|
| 主菜单（`_main_menu`） | 调用 `_on_exit()` 后退出程序 |
| 子模块菜单入口（`entry()`） | `break` 退出循环，返回主菜单 |
| 三级菜单 / 子流程函数 | `return` 返回调用方（二级菜单继续循环） |
| `confirm` / `text_input` 内部 | 抛 `UserCancel`，由调用方 `except UserCancel: return` |

**规则**：所有 `menu.py` 的 `entry()` 函数必须用 `try/except UserCancel: break` 包裹主循环中的 `select()` 调用；子流程函数用 `try/except UserCancel: return`。禁止在 `entry()` 以外传播 `UserCancel` 到主菜单层（主菜单自己处理）。

### 清屏规范

#### 公共函数

```python
# core/prompt.py

def clear_screen() -> None:
    """清除终端屏幕（跨平台）"""
    os.system('cls' if os.name == 'nt' else 'clear')
```

**铁律：所有模块禁止直接写 `os.system('cls' if os.name == 'nt' else 'clear')`，必须调用 `clear_screen()`。**

#### 清屏时机规范

| 函数类型 | 清屏时机 | 原因 |
|---------|---------|------|
| 直接输出型（无用户输入） | 函数开头调用 `clear_screen()` | 进入即清屏，保证干净界面 |
| 有用户输入型（`text_input` / `select` / `confirm`） | **交互返回后**调用 `clear_screen()`，再输出结果 | `text_input`/`select`/`confirm` 内部会渲染 breadcrumb，提前清屏会被覆盖 |

```python
# ❌ 错误：清屏在 text_input 前，会被 text_input 渲染覆盖
def show_ping():
    clear_screen()           # 被 text_input 渲染覆盖，无效
    host = text_input(...)
    console.print(result)

# ✅ 正确：清屏在 text_input 返回后、输出结果前
def show_ping():
    host = text_input(...)   # text_input 渲染自己的 breadcrumb
    clear_screen()           # text_input 返回后清屏
    console.print(result)    # 输出在干净屏幕上
```

### Ctrl+C 处理规范（完整）

#### entry() 层统一捕获

所有模块的 `entry()` 函数中，`show_*` 调用必须被 `try/except KeyboardInterrupt` 包裹，确保子功能中按 Ctrl+C 只回到该模块菜单，而非退出到主菜单：

```python
def entry() -> None:
    while True:
        try:
            key = select(...)
        except UserCancel:
            break
        if key is None:
            break

        # ✅ 所有 show_* 调用外包裹 KeyboardInterrupt
        try:
            if key == "1":
                show_foo()
            elif key == "2":
                show_bar()
        except KeyboardInterrupt:
            pass   # 吞掉，继续 while True → 回到子菜单
```

#### 流式输出命令（streaming）

对于 `traceroute`、`tail_follow` 等流式输出命令，Ctrl+C 中断后直接 `return`，不调用 `pause()`：

```python
# ✅ 正确：Ctrl+C → return 直接返回上级菜单
try:
    for line in traceroute(host):
        console.print(line)
except KeyboardInterrupt:
    return      # 直接返回，不 pause()
pause()         # 只在正常完成后执行
```

#### 子进程 Ctrl+C 隔离（Windows）

使用 `subprocess.Popen` / `subprocess.run` 启动子进程（ping / tracert 等）时，必须加 `CREATE_NEW_PROCESS_GROUP` 防止 Ctrl+C 同时发给父子进程：

```python
# network/commands.py

kwargs = {}
if sys.platform == "win32":
    kwargs["creationflags"] = subprocess.CREATE_NEW_PROCESS_GROUP
process = subprocess.Popen(cmd, ..., **kwargs)

# subprocess.run 同理
result = subprocess.run(cmd, ..., **kwargs)
```

### pause() 实现规范

`pause()` 必须能抵抗残余的 Ctrl+C 信号（子进程中断后信号可能仍在传播中）：

```python
def pause(msg: str = '') -> None:
    """等待任意键，msg 默认读 t('prompt.pause')。"""
    # Windows: 用 ctypes.SetConsoleCtrlHandler 彻底屏蔽 Ctrl+C
    # Unix:    用 signal.SIG_IGN 屏蔽 SIGINT
    # 内层 _read_key() 外必须 try/except (KeyboardInterrupt, EOFError, OSError)
    # 任何异常都静默吞掉，不向外传播
```

**关键**：Windows 上 `signal.SIG_IGN` 不能阻止 `msvcrt.getwch()` 收到 Ctrl+C，必须使用 `ctypes.windll.kernel32.SetConsoleCtrlHandler` 注册自定义处理器（返回 1 表示吞掉所有控制台事件），在 `finally` 中反注册。

### spinner() 信号屏蔽规范

Rich `Progress`（spinner 底层）使用后台渲染线程。Windows 上 Ctrl+C 发出的 `GenerateConsoleCtrlEvent` 会在渲染线程持锁期间命中进程，导致 Python 来不及 `raise KeyboardInterrupt` 就崩溃。因此 **spinner 生命周期内必须在 OS 层屏蔽 Ctrl+C**：

```python
# core/progress.py — spinner()

@contextmanager
def spinner(msg: str) -> Iterator[None]:
    _interrupted = False

    # Windows: SetConsoleCtrlHandler 吞掉 Ctrl+C + signal.SIG_IGN
    # Unix:    自定义 SIGINT handler 只记录标志位
    # → spinner 内部完全不会收到 KeyboardInterrupt

    try:
        with Progress(...) as p:
            yield
    finally:
        # 恢复原始信号处理器

    if _interrupted:
        raise KeyboardInterrupt  # 在主线程安全抛出
```

**规则**：所有耗时操作（subprocess / httpx / socket）如果在 `spinner` 内执行，Ctrl+C 由 spinner 统一延迟处理，不需要在调用方额外屏蔽信号。

### _ctrlc_guard() 即时响应方案

`spinner()` 的 `shield_ctrlc` 只能等操作完成后再抛出异常，对于阻塞时间很长的操作（`tracert` 超时节点每跳等 3s、`httpx` 网络请求等），用户按 Ctrl+C 后感知到的延迟不可接受。

**解决方案**：`network/menu.py` 内定义 `_ctrlc_guard()` 上下文管理器，在 Ctrl+C 时**立即**执行以下操作：

```python
# network/menu.py — _ctrlc_guard()

@contextmanager
def _ctrlc_guard(kill_ref=None, cancel_event=None):
    # Windows: SetConsoleCtrlHandler 返回 1 吞掉 Ctrl+C
    # 同时立即：cancel_event.set() + kill_ref[0].kill()
    # Unix: 自定义 SIGINT handler 执行同样操作
    yield interrupted  # interrupted[0] = True 表示被中断
    # finally: 恢复原始信号处理器
```

**配套 API 扩展**（`network/commands.py`）：

| 函数 | 新增参数 | 作用 |
|---|---|---|
| `ping()` | `kill_ref: list` | 写入 Popen 对象，供外部 kill() |
| `traceroute()` | `kill_ref: list` | 写入 Popen 对象，供外部 kill() |
| `speed_test_download()` | `cancel_event: Event` | 每次 chunk 后检查，立即停止下载 |
| `get_public_ip()` | `cancel_event: Event` | 每次 API 请求前检查，立即放弃 |

**使用模式**：

```python
# 带 subprocess 的操作（ping / traceroute）
kill_ref: list = []
with _ctrlc_guard(kill_ref=kill_ref) as interrupted:
    with spinner("..."):
        result = ping(host, kill_ref=kill_ref)
if interrupted[0]:
    return  # 立即返回上级菜单

# 带网络请求的操作（speed_test / public_ip）
cancel_event = threading.Event()
with _ctrlc_guard(cancel_event=cancel_event) as interrupted:
    with spinner("..."):
        result = get_public_ip(cancel_event=cancel_event)
if interrupted[0]:
    return

# 流式输出（traceroute）
kill_ref: list = []
with _ctrlc_guard(kill_ref=kill_ref) as interrupted:
    for line in traceroute(host, kill_ref=kill_ref):
        console.print(line)
if interrupted[0]:
    return
```

**spinner(raw=True) 规范**：`_ctrlc_guard` 内使用 `spinner` 时**必须**传 `raw=True`，否则 `spinner` 内部的 `shield_ctrlc()` 会覆盖 `_ctrlc_guard` 注册的信号处理器，导致 `kill_ref[0].kill()` / `cancel_event.set()` 永远不会被调用。

**验证结果**：全部 6 项探针实测 Ctrl+C 额外延迟：Ping 16ms、路由追踪 0ms、DNS 自然完成、端口扫描自然完成、下载测速 234ms、公网 IP 16ms。

### 强制规则

1. **所有 .py 文件禁止出现硬编码颜色**（`'bold green'`、`'#ff5f5f'` 等），必须通过 `get_color()` 获取
2. **所有 .py 文件禁止出现硬编码图标 emoji**，必须通过 `get_icon()` 获取

4. **theme.py 是唯一的样式出口**，其他模块不直接读 YAML

## 插件式模块架构

### 核心理念

每个功能模块（软件管理、系统监控、防火墙……）都是一个**独立插件**，与 main.py 和其他模块零耦合。

- main.py **不 import 任何模块包**，只通过注册表动态发现
- 模块之间**禁止互相 import**，只能依赖 core/
- 新增模块 = 新建目录 + 实现 `register()` 函数，无需改动任何已有文件
- 删除模块 = 删除目录，主菜单自动消失，不报错

### 注册协议

每个模块包的 `__init__.py` 必须暴露一个 `register()` 函数，返回 `ModuleInfo`：

```python
# core/module.py — 模块注册协议

@dataclass
class ModuleInfo:
    key: str             # 模块唯一标识，如 'software' / 'monitor'
                         # 菜单显示名由 get_icon(key) + t(f'menu.{key}') 动态组合
    description_key: str # i18n key，如 'module.software.desc'
    order: int           # 排序权重（越小越靠前）
    entry: Callable      # 入口函数（点击菜单项后调用）
    platforms: list[str] # 支持的平台：['linux', 'windows', 'darwin']
    enabled: bool        # 是否启用（可通过配置文件动态关闭）
```

```python
# 示例：monitor/__init__.py

from core.module import ModuleInfo

def register() -> ModuleInfo:
    from monitor.menu import menu_main
    return ModuleInfo(
        key='monitor',
        description_key='module.monitor.desc',
        order=20,
        entry=menu_main,
        platforms=['linux', 'windows', 'darwin'],
        enabled=True,
    )
```

### 自动发现机制

```python
# core/loader.py — 模块加载器

def discover_modules() -> list[ModuleInfo]:
    """
    双模式加载：

    1. 开发模式（源码运行）：
       - 扫描项目根目录下所有包
       - 找到含 register() 函数的包
       - 调用 register() 收集 ModuleInfo

    2. 打包模式（单文件）：
       - 读取 _registry.py 静态注册表（build.py 打包时自动生成）
       - _registry.py 格式：MODULE_LIST = [('software', 'software'), ('monitor', 'monitor'), ...]
       - 逐个 import + 调用 register()

    3. 通过 config._is_frozen() 自动判断使用哪种模式

    最终：按 order 排序，过滤当前平台不支持的模块 + 未启用的模块。
    """
```

main.py 只做：

```python
modules = discover_modules()
choices = [f'{get_icon(m.key)} {t(f"menu.{m.key}")}' for m in modules]
# select() 后调用 m.entry()
```

### 模块内部分层

每个模块内部统一遵循三层结构，互不越界：

```
{module}/
├── __init__.py      # 只暴露 register()
├── menu.py          # UI 层：菜单交互，调用 commands
└── commands.py      # 逻辑层：纯业务，调用 core/ 工具
```

- **menu.py** — 只负责菜单渲染和用户交互，调用 commands.py
- **commands.py** — 只负责业务逻辑，调用 core/runner.py、core/platform.py 等
- **禁止** commands.py 调用 menu.py（单向依赖，UI → 命令，不可反向）
- **禁止** 模块 A 的 commands.py import 模块 B 的任何内容

### 软件管理的 Recipe 插件

software/ 模块内部再有一层插件机制：每个软件是一个独立 recipe：

```
software/
├── __init__.py
├── menu.py
├── commands.py
├── registry.py          # recipe 自动发现 + 注册
├── base.py              # Recipe 抽象基类
└── recipes/
    ├── __init__.py
    ├── docker.py         # 继承 Recipe，实现 install / uninstall / detect
    ├── nginx.py
    └── ...               # 新增软件 = 新增一个 .py，无需改其他文件
```

```python
# software/base.py — 安装配方抽象基类

class Recipe(ABC):
    key: str                # 唯一标识，如 'docker' / 'nginx'
                            # 显示名由 get_icon(key) + t(f'software.{key}') 动态组合
    platforms: list[str]    # 支持平台
    dependencies: list[str] = []  # 依赖的其他 Recipe key，安装前自动检查并提示安装

    # ── 能力声明（子类可覆盖）──
    has_upgrade: bool = True       # 是否支持升级（False 时菜单中灰显）
    has_diagnose: bool = False     # 是否支持诊断（True 时菜单显示 (4) 诊断）
    has_manage: bool = False       # 是否支持管理（True 时菜单显示 (5) 管理，如 WireGuard peer 管理）
    has_submenu: bool = False      # 是否有子菜单（如 WireGuard → 公网/内网）
    has_wizard: bool = False       # 是否使用自定义安装向导
    hidden: bool = False           # 是否在分类/搜索列表中隐藏

    @abstractmethod
    def detect(self) -> str | None:
        """检测是否已安装，返回版本号或 None"""

    @abstractmethod
    def versions(self) -> list[str]:
        """
        可安装的版本列表，双源策略：

        1. 内置列表：类属性 BUILTIN_VERSIONS 硬编码 LTS 版本号（离线可用）
        2. 在线获取：从官方 API 拉取最新版本
           - Docker:  https://api.github.com/repos/docker/cli/tags
           - 各 Recipe 自行实现
        3. 优先在线，超时（3s）或失败回退到内置列表
        """

    @abstractmethod
    def install(self, version: str) -> bool:
        """执行安装，返回是否成功"""

    @abstractmethod
    def uninstall(self) -> bool:
        """执行卸载，返回是否成功"""

    def diagnose(self) -> None:
        """运行诊断检查（has_diagnose=True 时子类实现）"""

    def manage(self) -> None:
        """运行管理界面（has_manage=True 时子类实现，如 WireGuard peer 增删查）"""
```

## 核心设计原则

### 解耦规则（强制）

```
依赖方向（单向，禁止反向或横向）：

main.py
  │
  ▼
core/loader.py ──discover_modules()──→ 扫描所有插件包
  │
  ▼
core/*（theme / prompt / runner / config / platform / privilege / progress）
  ▲               ▲               ▲
  │               │               │
software/    monitor/    service/    firewall/    ...（各插件模块）
  │
  ▼
software/recipes/*（二级插件，仅依赖 software/base.py + core/）
```

**铁律：**

1. **main.py → core/ 只**：main.py 只 import core/loader，不 import 任何插件包
2. **插件 → core/ 只**：所有插件模块只 import core/ 下的公共工具
3. **插件 ↛ 插件**：模块之间禁止任何 import，即 software/ 不能 import monitor/
4. **menu → commands 只**：模块内 menu.py 调用 commands.py，反向禁止
5. **recipe → base 只**：recipes/*.py 只继承 software/base.py + 调用 core/
6. **配置驱动开关**：config/common.yaml 中 `modules.{name}.enabled` 控制模块是否加载

**违反检测**（CI / pre-commit 可加 lint 规则）：

- 插件包内不允许出现 `from software import` / `from monitor import` 等跨模块 import
- commands.py 不允许 `from .menu import`

### 全自动安装

1. **环境探测** — 自动识别 OS 发行版、包管理器、init 系统、架构
2. **镜像源** — 国内自动切换清华/阿里源，海外用官方源
3. **依赖解析** — 安装前检查依赖，缺失自动补装
4. **步骤进度** — 每一步有编号和描述，Rich Progress 实时展示
5. **异常自愈** — 网络超时重试、锁冲突等待、源不可用自动换源
6. **安装验证** — 完成后自动 `--version` 验证，失败回滚
7. **幂等性** — 已安装则提示版本信息，不重复安装

### 全自动卸载

1. **服务停止** — 卸载前自动停止关联服务
2. **完整清理** — 包 + 配置 + 数据（数据清理需二次确认）
3. **依赖检查** — 不卸载其他软件依赖的包
4. **步骤进度** — 同安装

## 国际化（i18n）

### 设计目标

仅支持**中文（zh）**和**英文（en）**两种语言，启动时自动检测系统语言，用户也可通过菜单或配置文件手动切换。

### 实现方案

```
core/
├── i18n.py              # 国际化引擎
└── locale/
    ├── zh.yaml          # 中文文案
    └── en.yaml          # 英文文案
```

#### core/i18n.py 核心逻辑

```python
_current: dict[str, str] = {}
_lang: str = 'en'

def init() -> None:
    """
    语言决策优先级（从高到低）：
    
    1. config/common.yaml → language 字段
       - 值为 'zh' / 'en' → 直接使用（用户曾手动切换过）
       - 值为 'auto' 或字段不存在 → 进入第 2 步
    
    2. 检测系统语言
       - Linux: 读 LANG / LC_ALL 环境变量
         → 'zh_CN.UTF-8' → zh
         → 'en_US.UTF-8' → en
       - Windows: ctypes.windll.kernel32.GetUserDefaultUILanguage()
         → 0x0804 → zh
       - macOS: defaults read -g AppleLanguages
         → ['zh-Hans'] → zh
    
    3. 兜底默认 → en
    
    决策结果加载对应 locale/{lang}.yaml 到 _current。
    """

def t(key: str, **kwargs) -> str:
    """翻译：t('install.progress', step=2, total=5) → '[2/5] 安装中...'"""
    return _current.get(key, key).format(**kwargs)

def switch(lang: str) -> None:
    """
    切换语言并持久化：
    1. 重新加载 locale/{lang}.yaml → _current
    2. 写入 config/common.yaml → language: {lang}
    3. 下次启动时 init() 读到 language: zh/en → 直接使用，不再自动检测
    """

def current_lang() -> str:
    """返回当前语言代码：'zh' / 'en'"""
    return _lang
```

#### 配置统一管理

语言设置存储在 config/common.yaml，与其他配置统一管理：

#### 切换流程

```
首次启动
 │
 ├─ 读 config/common.yaml → language 字段
 │   └─ 值为 'auto' 或不存在 → 检测系统语言 → 得到 'zh' 或 'en'

用户手动切换
 │
 ├─ 主菜单 → (l) 🌐 Language / 语言

 │   ├─ (2) English
 │   └─ (0) 🔙 返回

 │
 └─ 下次启动：
     读到 language: zh → 直接加载中文，不再检测系统语言
```

## 智能源管理（Mirror Manager）

### 设计目标

用户**零配置**，启动后自动选择最优源；下载过程中遇到故障**无感切换**，不中断进度。

### 架构

```
core/
├── mirror.py            # 智能源管理器
└── mirrors/
    ├── schema.py        # MirrorSource 数据类
    └── sources.yaml     # 所有源地址清单（内置，随版本更新）
```

### 核心流程

```
启动 → 检测 IP 归属地 → 加载源清单 → 按优先级排序
                ↓
        ┌───────────────┐
        │  国内 IP？     │
        │  是 → 国内源组 │
        │  否 → 国外源组 │
        └───────────────┘
                ↓
        对源组内所有源并发测速（HEAD 请求，取响应最快的 3 个）
                ↓
        缓存排序结果到 {data}/cache/mirror_cache.yaml（有效期 24h）
                ↓
        下载时使用最优源，失败 → 自动切换下一个源
```

### IP 归属地检测

```python
# core/mirror.py

def detect_region() -> str:
    """
    检测当前网络是否在中国大陆，返回 'cn' 或 'intl'。
    
    策略（按优先级）：
    1. 配置文件手动指定 region: cn / intl → 直接使用
    2. 请求 https://ipinfo.io/json（超时 3s）→ 读 country 字段
    3. 备用 https://ip.useragentinfo.com/json → 读 country
    4. 以上全失败 → 尝试 ping mirrors.aliyun.com，延迟 < 50ms 判定为 cn
    5. 兜底默认 intl（使用官方源更安全）
    """
```

### 无感切换 + 断点续传

```python
def download_with_failover(
    url_template: str,     # 含 {mirror} 占位符的 URL 模板
    dest: Path,
    mirrors: list[str],    # 已排序的源列表
    chunk_size: int = 65536,
) -> bool:
    """
    带自动降级的下载器：
    
    1. 用 mirrors[0] 开始下载
    2. 如果连接超时（5s）或下载速度 < 10KB/s 持续 10s → 切换 mirrors[1]
    3. 利用 HTTP Range 头实现断点续传，已下载部分不重复下载
    4. Rich Progress 进度条持续显示，源切换时更新提示但不重置进度
    5. 所有源都失败 → 返回 False
    """
```

## 自动更新机制

### 设计目标

OpsKit 最终打包为**单文件可执行程序**（opskit / opskit.exe），启动后自动检测更新，全程静默完成，用户无感知、无焦虑。

### 文件布局

```
# Linux / macOS
/usr/local/bin/opskit            ← 可执行文件（唯一文件）
/var/lib/opskit/                 ← 数据目录（用户数据 + 配置 + 备份）
├── config/common.yaml           ← 用户配置
├── data/                        ← 用户数据（安装记录、备份历史等）
├── cache/                       ← 缓存（源测速、更新包）
├── logs/                        ← 运行日志
└── backups/                     ← 自动备份

# Windows
C:\Program Files\opskit\opskit.exe
C:\ProgramData\opskit\           ← 同上结构
```

**关键设计：可执行文件与用户数据完全分离，更新只替换可执行文件，永不触碰数据目录。**

### 更新流程

```
启动
 │
 ├─ 后台线程：检查更新（不阻塞主菜单）
 │   │
 │   ├─ GET https://api.github.com/repos/ougato/opskit/releases/latest
 │   │   解析 Assets → 匹配 os + arch → 得到 { version, url, sha256, size, changelog }
 │   │
 │   ├─ 比较 current_version vs latest_version
 │   │   └─ 相同 → 结束
 │   │   └─ 不同 → 继续
 │   │
 │   ├─ 下载新版本到临时文件（{cache}/opskit_v{ver}.tmp）
 │   │   └─ 使用 mirror 智能源 + 断点续传
 │   │   └─ 下载失败 → 静默跳过，下次启动重试
 │   │
 │   ├─ 校验 SHA256
 │   │   └─ 不匹配 → 删除临时文件 → 日志告警 → 跳过
 │   │
 │   ├─ 通知用户（非阻塞提示条）：
 │      "🔄 新版本 v{n} 已下载，退出时自动更新"
 │   │
 │   └─ 用户退出时 → 执行热替换
 │
 └─ 正常进入主菜单（更新检查不影响使用）
```

### 热替换（核心难点）

```python
# core/updater.py

def hot_replace(new_binary: Path, current_binary: Path) -> None:
    """
    关键步骤：
    
    1. 备份当前可执行文件
       - Linux/macOS: cp opskit → {cache}/opskit_v{old_ver}.bak
       - Windows: copy opskit.exe → {cache}\opskit_v{old_ver}.bak
    
    2. 替换可执行文件
       - Linux/macOS: mv new → current（原子操作，同文件系统下 rename 是原子的）
       - Windows: 无法覆盖正在运行的 exe！
         → 方案：写一个 .bat / .ps1 脚本：
           (1) 等待当前进程退出（taskkill /PID {pid} /F 兜底）
           (2) del opskit.exe
           (3) move opskit_new.exe opskit.exe
           (4) start opskit.exe（可选，如用户选择自动重启）
           (5) del 自身 .bat
    
    3. 验证新版本可用
       - 启动新版本 opskit --version，检查退出码
       - 失败 → 自动回滚（mv bak → current）
    
    4. 清理备份和临时文件
    """
```

### 已知坑点与对策

| 坑点 | 描述 | 对策 |
|------|------|------|
| **Windows 文件锁** | 正在运行的 exe 无法被覆盖或删除 | 退出后用外部 .bat/.ps1 脚本完成替换 |
| **权限不足** | 程序安装在 /usr/local/bin 或 Program Files 需要 root/admin | 更新时自动提权（sudo / runas），失败则提示用户 |
| **下载中断** | 网络不稳定导致更新包下载不完整 | SHA256 校验 + 断点续传 + 多源降级 |
| **新版本崩溃** | 更新后新版本无法启动 | 启动时自动验证（--version），失败立即回滚到备份 |
| **回滚也失败** | 备份文件损坏或被删除 | 备份后立即验证备份文件完整性；保留最近 2 个版本 |
| **更新服务器挂了** | 无法获取版本信息 | 静默跳过，不影响正常使用；下次启动重试 |
| **并发操作** | 用户同时运行多个 opskit 实例 | 全局文件锁（`{data}/cache/opskit.lock`）：启动时获取排他锁（Linux fcntl / Windows msvcrt），已有实例运行时提示退出；同时覆盖更新和安装操作 |
| **版本降级** | 用户误装旧版本 | 支持选择版本安装，降级时警告但允许 |
| **磁盘空间不足** | 下载或备份时磁盘满 | 下载前检查剩余空间（需要 2x 文件大小） |
| **杀毒误报** | UPX 压缩或自更新行为触发杀毒软件 | Windows 签名（可选）+ 提供未压缩版本 |

### 更新配置

```yaml
# config/common.yaml
update:
  enabled: true               # 是否启用自动更新
  channel: stable             # stable / beta
  check_interval: 86400       # 检查间隔（秒），默认 24h
  auto_apply: true            # 退出时自动应用，false 则仅提示
  repo: ougato/opskit         # GitHub 仓库（用于 Releases API）
  mirrors:                    # 更新包下载源（国内加速，走智能源管理）
    - https://mirror.ghproxy.com/github.com/ougato/opskit/releases/download
    - https://github.com/ougato/opskit/releases/download
```

**更新服务端方案**：直接使用 GitHub Releases API，零维护成本：
- 版本检测：`GET https://api.github.com/repos/{repo}/releases/latest`
- 下载链接：Release Assets（每个 OS + arch 一个文件）
- SHA256 校验：Release Body 中包含校验值
- 国内加速：通过 mirrors 列表中的代理下载

**GitHub API 速率限制应对**：
- 未认证请求每小时 60 次，多台机器共用出口 IP 时容易触限
- 应对策略：
  1. 严格遵守 `check_interval`（默认 24h），将上次检查时间写入 `{data}/cache/update_check.json`
  2. 收到 HTTP 403 / 429 时静默跳过，下次启动重试
  3. 可选配置 `update.github_token`（认证后 5000 次/小时）
  4. 响应头含 `X-RateLimit-Remaining`，剩余 < 5 时主动延迟下次检查

**国内代理可靠性说明**：
- mirrors 列表中的社区代理（如 ghproxy）可能随时关闭，不保证可用性
- updater.py 会逐个尝试 mirrors 列表，全部失败回退到 GitHub 原生地址
- 用户可在 config/common.yaml 中自定义 mirrors 列表，添加自己信任的代理

### 安装选择版本

用户安装 OpsKit 时可以选择版本：

```
$ opskit install --version 5          # 安装指定版本
$ opskit install                      # 安装最新版本（默认）
$ opskit update                       # 更新到最新版本
$ opskit update --version 8           # 更新到指定版本
$ opskit rollback                     # 回滚到上一个版本
```

### 安装/更新/卸载流程中的坑点

#### 更新 = 下载新版 + 替换旧版（不是卸载再安装）

**绝对不做「先卸载再安装」**，原因：

1. 卸载可能失败 → 用户变成既没有旧版也没有新版
2. 卸载会中断用户正在使用的功能
3. 卸载触发数据清理风险

正确做法：**直接覆盖二进制文件**，配置和数据完全不动。

#### 万一覆盖失败的兜底

```
更新失败兜底链：

1. 原子替换失败 → 重试 1 次
2. 重试失败 → 保留旧版本不动，提示下次重试
3. 旧版本损坏 → 从 {cache}/opskit_v{n}.bak 恢复
4. 备份也丢了 → 提示用户手动下载：opskit download --version {n}
5. 全部失败 → 日志记录完整错误链，提示用户联系支持
```

### 软件管理中的版本选择

用户通过 OpsKit 安装第三方软件时，也支持版本选择：

## 用户数据安全保护

### 核心原则

**用户数据神圣不可侵犯**——更新、降级、卸载任何操作都不能丢失用户数据。

### 数据分类

| 类别 | 路径 | 更新时 | 卸载时 |
|------|------|--------|--------|
| **可执行文件** | /usr/local/bin/opskit | ✅ 替换 | ✅ 删除 |
| **用户配置** | {data}/config/ | ❌ 不动 | ⚠️ 保留（除非用户主动选择清除） |
| **安装记录** | {data}/data/registry.yaml | ❌ 不动 | ⚠️ 保留 |
| **备份数据** | {data}/backups/ | ❌ 不动 | ⚠️ 保留 |
| **日志** | {data}/logs/ | ❌ 不动 | ⚠️ 可选清除 |
| **缓存** | {data}/cache/ | ❌ 不动 | ✅ 可安全删除 |

### 更新时的数据保护

```python
def safe_update(new_binary: Path) -> bool:
    """
    1. 备份当前可执行文件 → {cache}/opskit_v{ver}.bak
    2. 备份当前配置 → {backups}/config_{timestamp}.yaml
    3. 替换可执行文件（只替换这一个文件）
    4. 验证新版本启动正常
    5. 验证配置兼容性（新版本读取旧配置不报错）
    6. 失败 → 回滚可执行文件 + 配置
    """
```

### 卸载时的数据保护

```
OpsKit 卸载流程：

1. 提示用户：
   "⚠️ 即将卸载 OpsKit，以下数据将保留：
    - 用户配置（{data}/config/）
    - 安装记录（{data}/data/）
    - 备份数据（{data}/backups/）
    
    是否同时清除以上数据？"
    
   (1) 仅卸载程序，保留数据（推荐）
   (2) 卸载程序并清除所有数据
   (3) 取消

2. 选择 (1)：只删除可执行文件，数据完整保留
   → 重新安装后数据自动恢复

3. 选择 (2)：二次确认
   "⚠️ 此操作不可逆，确认删除所有数据？[y/N]"
   → 确认后先创建最终备份到用户桌面 → 再清除
```

### 配置迁移

新版本可能修改配置格式，需要自动迁移：

```python
# core/config.py

def migrate_config(config: dict, from_version: int, to_version: int) -> dict:
    """
    配置迁移策略：
    
    1. 每个版本定义一个 migration 函数
    2. 按版本链逐步迁移：1 → 2 → 3
    3. 迁移前自动备份原配置
    4. 迁移失败 → 使用默认配置 + 保留原配置为 .bak
    5. 只增不删：新版本只添加新字段默认值，不删除旧字段
    """
```

### 配置路径双模式

开发时（源码运行）和打包后（单文件）配置路径不同，`config.py` 自动判断：

```python
# core/config.py

    """判断是否为打包后的单文件运行模式"""
    return getattr(sys, 'frozen', False) or '__compiled__' in globals()

    数据目录解析优先级：

    1. 环境变量 OPSKIT_DATA_DIR（调试/测试用）
    2. 打包模式 → 系统数据目录
       - Linux:  /var/lib/opskit/
       - macOS:  ~/Library/Application Support/opskit/
       - Windows: C:\ProgramData\opskit\
    3. 开发模式 → 项目根目录
       - 即 main.py 所在目录
    """

def get_config_path() -> Path:
    """返回 get_data_dir() / 'config' / 'common.yaml'"""

def ensure_config() -> dict:
    """
    首次运行时自动创建默认配置：

    1. 检查 get_config_path() 是否存在
    2. 不存在 → 从内置默认配置模板生成（DEFAULT_CONFIG dict 定义在 constants.py）
    3. 自动创建父目录（makedirs(exist_ok=True)）
    4. 写入 YAML + 返回配置 dict
    5. 已存在 → 加载并返回，同时检查是否需要迁移
```

资源文件（themes / locale / mirrors）读取逻辑类似：
- 打包模式：通过 `importlib.resources` 或 `sys._MEIPASS`（PyInstaller）读取嵌入资源
- 开发模式：直接从项目 `core/themes/` 等目录读取

## 单文件最小化打包

### 设计目标

最终产物为**单个可执行文件**（无外部依赖、无解压目录），体积目标：

| 平台 | 目标大小 | 上限 |
|------|---------|------|
| Linux x86_64 | < 12 MB | 18 MB |
| Windows x86_64 | < 14 MB | 20 MB |
| macOS x86_64 / arm64 | < 13 MB | 18 MB |

> 注：psutil 含 C 扩展约 +2-3MB，已计入体积估算。psutil 采用懒加载（仅进入监控菜单才 import），打包时仍需包含但不影响启动速度。

### 为什么能做到小

1. **纯脚本项目** — 无图片、无前端资源、无 ML 库
2. **依赖极少** — 仅 Rich + Typer + PyYAML + httpx + psutil，无重量级框架
3. **单文件模式** — 打包器将所有 .pyc + 运行时压缩为一个 binary

### 打包方案对比

| 方案 | 大小 (Hello World) | 大小 (Rich CLI) | 启动速度 | 跨平台 | 推荐 |
|------|-------------------|-----------------|---------|--------|------|
| **Nuitka --onefile** | ~3.8 MB | ~8-12 MB | ⚡ 快 | ✅ | ✅ 首选 |
| PyInstaller --onefile | ~6 MB | ~15-20 MB | 🐢 较慢 | ✅ | 备选 |
| PyInstaller + UPX | ~4 MB | ~10-14 MB | 🐢 较慢 | ⚠️ | 备选 |

### 最小化策略

#### 1. 干净虚拟环境

```bash
# 每次打包使用全新 venv，只安装必要依赖
python -m venv build_env
pip install rich typer pyyaml httpx psutil
# 不安装 dev 依赖（pytest / black / mypy 等）
```

#### 2. Nuitka 编译优化（首选）

```bash
nuitka \
  --onefile \
  --python-flag=no_site \
  --python-flag=no_docstrings \
  --python-flag=-OO \
  --remove-output \
  --assume-yes-for-downloads \
  --include-data-dir=core/themes=core/themes \
  --include-data-dir=core/locale=core/locale \
  --include-data-dir=core/mirrors=core/mirrors \
  --include-package=rich._unicode_data \
  --include-module=_registry \
  --include-package=software \
  --include-package=monitor \
  --include-package=network \
  --include-package=service \
  --include-package=backup \
  --include-package=firewall \
  --include-package=loganalysis \
  --include-package=security \
  --output-filename=opskit \
  --company-name=Ougato \
  --product-name=OpsKit \
  --file-version=$VERSION \
  main.py
```

关键参数说明：
- `--onefile` — 单文件输出
- `--python-flag=no_docstrings` — 去除 docstring，减小体积
- `--python-flag=-OO` — 最高优化级别，去除 assert + docstring
- `--python-flag=no_site` — 不导入 site 包，加速启动

#### 3. PyInstaller 备选（+ UPX）

```bash
# Linux / macOS——分隔符用“:”
pyinstaller \
  --onefile \
  --strip \
  --noupx \
  --add-data "core/themes:core/themes" \
  --add-data "core/locale:core/locale" \
  --add-data "core/mirrors:core/mirrors" \
  --exclude-module tkinter \
  --exclude-module unittest \
  --exclude-module email \
  --exclude-module xml \
  --exclude-module pydoc \
  --exclude-module doctest \
  --name opskit \
  main.py

# Windows——分隔符用“;”
pyinstaller ^
  --onefile ^
  --noupx ^
  --add-data "core/themes;core/themes" ^
  --add-data "core/locale;core/locale" ^
  --add-data "core/mirrors;core/mirrors" ^
  --exclude-module tkinter ^
  --exclude-module unittest ^
  --exclude-module email ^
  --exclude-module xml ^
  --exclude-module pydoc ^
  --exclude-module doctest ^
  --name opskit ^
  main.py
```

> 注意：PyInstaller `--add-data` 分隔符在 Linux/macOS 是 `:`，Windows 是 `;`。`build.py` 中应用 `os.pathsep` 自动判断。Windows 不支持 `--strip`。

#### 4. UPX 压缩（可选，仅 Windows / Linux）

```bash
# 下载 UPX: https://github.com/upx/upx/releases
upx --best --lzma opskit        # Linux
upx --best --lzma opskit.exe    # Windows
# macOS 不建议用 UPX（签名 + SIP 问题）
```

UPX 注意事项：
- ✅ 可再压缩 30-50%
- ⚠️ 某些杀毒软件会误报 UPX 压缩的 exe
- ⚠️ macOS 使用 UPX 会破坏代码签名
- ⚠️ 增加少量启动时间（解压开销）

#### 5. 排除不需要的标准库模块

```python
# build.py 中维护排除列表
EXCLUDES = [
    'tkinter', 'unittest', 'email', 'xml', 'xmlrpc',
    'pydoc', 'doctest', 'test', 'distutils', 'setuptools',
    'lib2to3', 'ensurepip', 'idlelib', 'turtle', 'turtledemo',
    'multiprocessing',  # 如果不用多进程
    'sqlite3',  # 如果不用
    'ctypes.test',
]
```

#### 6. 懒加载 import

```python
# 非核心模块在函数内部 import，打包器可能据此减少打包范围
def menu_monitor():
    import psutil  # 只有进入监控菜单才加载
    ...
```

### 打包坑点

| 坑点 | 描述 | 对策 |
|------|------|------|
| **PyInstaller onefile 启动慢** | 每次启动需解压到临时目录 | 首选 Nuitka（编译为 C，无需解压） |
| **PyInstaller 临时目录残留** | 异常退出时 _MEI* 目录不会自动清理 | Nuitka 无此问题；PyInstaller 可加启动时清理 |
| **Nuitka 编译慢** | 首次编译需下载 C 编译器（MinGW / GCC） | CI 环境预装编译器；本地开发用 PyInstaller |
| **macOS 签名** | 未签名的二进制被 Gatekeeper 拦截 | CI 中集成 codesign（需 Apple 开发者账号） |
| **macOS 公证** | 10.15+ 未公证的二进制直接拒绝运行 | CI 加入 `xcrun notarytool submit`；或文档注明右键 → 打开 |
| **Windows Defender 误报** | 自更新 + UPX 压缩容易被标记 | 可选关闭 UPX；微软提交误报申诉 |
| **Linux strip 过度** | strip 可能移除必要符号导致段错误 | Nuitka 自动处理；PyInstaller 用 --strip 安全 |
| **隐式依赖遗漏** | 打包器未检测到动态 import 的模块 | 在 build.py 中显式声明 hidden imports |
| **Nuitka 遗漏 rich._unicode_data** | Rich 在运行时动态 `importlib.import_module('rich._unicode_data.unicode17-0-0')` 加载 Unicode 数据，Nuitka 的 `--nofollow` 模式无法追踪，导致打包后 `ModuleNotFoundError` 崩溃 | build.py 的 Nuitka 命令**必须**加 `--include-package=rich._unicode_data` |
| **Nuitka 遗漏插件模块** | 插件通过 `_registry.py` 动态 import，Nuitka 无法自动追踪 | build.py 的 Nuitka 命令**必须**为每个插件加 `--include-package={module}`（PyInstaller 用 `--hidden-import`） |
| **不同 glibc 版本** | 高版本 glibc 编译的二进制在旧系统跑不了 | 在 CentOS 7 / Ubuntu 18.04 等旧系统上编译 |

### 终端渲染与 Emoji 图标规则

#### Windows conhost.exe 兼容性（重要）

双击 exe 运行时，Windows 使用 `conhost.exe` 作为宿主终端（而非 Windows Terminal）。`conhost.exe` 对 emoji 渲染有严格限制：

| 类型 | Unicode 范围 | conhost 渲染 | 示例 |
|------|-------------|-------------|------|
| **4 字节 Supplementary Plane emoji** | U+1F000 以上 | ✅ 彩色 | 📦 🔼 🟢 🔴 |
| **2 字节 BMP emoji + U+FE0F** | U+0000-U+FFFF + FE0F | ❌ 单色/方块 | ⬆️ ▶️ ⏹️ ⚠️ ℹ️ |
| **2 字节 BMP emoji 无 FE0F** | U+0000-U+FFFF | ❌ 单色 | ⚡ ❌ ✅ |

**铁律**：

1. **主题 YAML 中所有图标必须使用 4 字节 Supplementary Plane emoji**（U+1F000 以上），禁止使用 BMP 范围 + `U+FE0F` 变体选择符的 emoji
2. 不可避免使用 BMP emoji 时（如 ❌ ✅），**必须去掉 `U+FE0F`**，接受单色渲染
3. 详细规则参见 `.windsurf/rules/emoji.md`

#### Windows VT 模式初始化（重要）

`main.py` 顶部必须在所有 import 之前启用 Windows Virtual Terminal Processing + UTF-8 代码页，否则 Rich 的 ANSI 转义序列在 `conhost.exe` 中不生效：

```python
# main.py 最顶部
if sys.platform == "win32":
    import ctypes, ctypes.wintypes
    # 启用 VT 模式（stdout + stderr）
    # 设置 UTF-8 代码页（65001）
    # reconfigure stdout/stderr encoding
```

**铁律**：此初始化代码必须在 `import typer` / `import rich` 之前执行，且不可删除。
